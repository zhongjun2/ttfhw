# CANN TTFHW 需求分析

**文档版本：** v1.0
**创建日期：** 2026-03-26
**状态：** 正式

---

## 一、背景

### 1.1 什么是 TTFHW

TTFHW（Time To First Hello World）是衡量开发者从零开始到第一次跑通核心功能所需时间的指标。

类比：
- TTFB（Time To First Byte）衡量网络响应速度
- TTFHW 衡量开发者体验速度

对 AI 计算框架而言，"Hello World" 等价于**第一个自定义算子跑通**——这是开发者能真正开始用这个框架做事的最低门槛。

### 1.2 为什么要测 CANN

Ascend CANN 是华为面向 AI 场景的计算架构，支持昇腾 NPU。相比 CUDA（NVIDIA）和 ROCm（AMD），CANN 的开发者生态尚在建设中，社区文档、工具链成熟度、社区响应速度等维度均缺乏客观的量化数据。

当前痛点：
- 没有客观数据说明 CANN 的开发者体验处于什么水平
- 不清楚哪个环节是最大的阻断点（文档可达性？镜像获取？工具链？社区响应？）
- 无法追踪 CANN 体验随版本迭代的变化趋势

### 1.3 项目定位

本项目构建一套针对 CANN 的 TTFHW 自动化测试框架，持续测量开发者体验的四个阶段，输出可量化、可对比、可追踪的指标数据。

## 二、目标用户

| 角色 | 使用方式 |
|------|---------|
| **CANN 开发者体验团队** | 获取客观数据，识别体验瓶颈，推动改善 |

## 三、测试范围

### 3.1 四个阶段

将开发者体验拆成四个连续阶段，每个阶段独立可测：

```
了解 → 获取 → 使用 → 贡献
Learn  Get    Use    Contribute
```

| 阶段 | 起点 | 终点 | 核心问题 |
|------|------|------|---------|
| **了解** | 什么都不知道，打开搜索引擎 | 找到官方安装文档，知道下一步怎么做 | 文档能不能被找到？找到了能不能看懂？ |
| **获取** | 知道要用 Docker 拉镜像 | CANN 环境在容器内可用，版本确认 | 镜像能不能拉下来？要几步？要多久？ |
| **使用** | CANN 环境就绪 | 第一个自定义算子编译并跑通 | 工具链能不能用？在没有硬件的情况下能不能跑？ |
| **贡献** | 遇到问题或有改进想法 | Issue 提交成功，CI 跑完 | 社区入口好不好找？响应速度怎么样？ |

### 3.2 测试方式

两种互补的测试方式并行运行：

| 方式 | 文件 | 目的 |
|------|------|------|
| **自动化测试** | `runner.py` + `stages/` | 高频执行，追踪时间趋势，机器可读指标 |
| **手动测试** | `docs/manual-test-procedure.md` | 模拟真实用户行为，捕捉自动化测试覆盖不到的体验细节 |

### 3.3 测试场景

**场景标识：** `zero-to-custom-op`

> 模拟一个从未接触过 CANN 的开发者，在没有物理 NPU 硬件的 x86_64 Linux 机器上，从零开始到第一个自定义算子跑通的完整过程。

**选择 AiCPU AddCustom 算子的原因：**
- 是 CANN 开发的最基础单元（自定义算子）
- AiCPU 模式在 CPU 上运行，不依赖物理 NPU，测试环境门槛低
- 官方文档有明确的 Hello World 示例

### 3.4 不在范围内

- 物理 NPU 硬件性能测试
- 多卡分布式训练测试
- 与具体 AI 模型的集成测试（MindSpore、PyTorch_NPU 等）
- 真实用户招募的 UX 测试

## 四、功能需求

### F1：了解阶段自动化

| 编号 | 需求 | 优先级 |
|------|------|--------|
| F1.1 | 调用搜索引擎搜索 CANN 相关关键词，找到官方文档入口 | P0 |
| F1.2 | 验证官方文档链接可访问（HTTP 200） | P0 |
| F1.3 | 记录官方链接在搜索结果中的排名 | P1 |
| F1.4 | 检测文档页面内的断链数量 | P1 |
| F1.5 | 记录从搜索到找到安装文档的跳转次数 | P1 |

### F2：获取阶段自动化

| 编号 | 需求 | 优先级 |
|------|------|--------|
| F2.1 | 执行 `docker pull` 拉取 CANN 镜像 | P0 |
| F2.2 | 启动容器，验证 CANN 工具链可用（`cann-info` 或等效命令） | P0 |
| F2.3 | 记录下载耗时、镜像大小 | P0 |
| F2.4 | 处理镜像拉取失败（DNS 失败、认证失败、超时），记录错误原因 | P0 |
| F2.5 | 容器使用后自动清理 | P1 |

### F3：使用阶段自动化

| 编号 | 需求 | 优先级 |
|------|------|--------|
| F3.1 | 在容器内编译 AiCPU AddCustom 算子 | P0 |
| F3.2 | 运行算子推理，验证输出误差 < 1e-5 | P0 |
| F3.3 | 记录编译耗时、运行耗时、编译错误/警告数量 | P0 |
| F3.4 | fixtures 算子代码版本与镜像版本锁定 | P1 |

### F4：贡献阶段自动化

| 编号 | 需求 | 优先级 |
|------|------|--------|
| F4.1 | 通过 Gitcode API 向 CANN 社区提交 Issue | P0 |
| F4.2 | 轮询检测 Issue 首次响应时间 | P0 |
| F4.3 | 向 fork 仓库推送代码，提交 PR | P1 |
| F4.4 | 监听 PR 的 CI 状态，分段记录排队/准备/运行时间 | P1 |
| F4.5 | token 不配置时提前失败，不发起 API 请求 | P0 |

### F5：报告输出

| 编号 | 需求 | 优先级 |
|------|------|--------|
| F5.1 | 输出 JSON 格式报告（机器可读，含所有指标字段） | P0 |
| F5.2 | 输出 Markdown 格式报告（人类可读，含摘要和异常说明） | P0 |
| F5.3 | 每个阶段有 `pass` / `warn` / `fail` 三态状态 | P0 |
| F5.4 | 每个阶段有 `errors` 数组，记录所有错误信息 | P0 |

### F6：接口

| 编号 | 需求 | 优先级 |
|------|------|--------|
| F6.1 | 提供 MCP 工具接口（`cann_ttfhw_run_all`、`cann_ttfhw_run_stage`、`cann_ttfhw_report`） | P1 |
| F6.2 | 提供 Claude Skill 命令（`/cann-ttfhw` 等） | P1 |
| F6.3 | 单阶段可独立运行，不依赖其他阶段先执行 | P0 |

## 五、非功能需求

| 编号 | 需求 | 说明 |
|------|------|------|
| NF1 | **可重复性** | 相同环境下多次运行结果可对比；使用 Docker 标准化环境 |
| NF2 | **可扩展性** | Stage 接口统一，新增其他社区（ROCm、CUDA）只需实现同一套接口 |
| NF3 | **幂等性** | Issue/PR 标题含时间戳，避免重复提交冲突 |
| NF4 | **安全性** | token 等敏感信息通过环境变量注入，不硬编码进代码或报告 |
| NF5 | **容错性** | 某阶段失败默认继续执行后续阶段（可配置为终止），不因单点失败丢失其他数据 |
| NF6 | **可配置** | 镜像版本、超时时间、token、搜索引擎等通过 `config.yaml` 配置，不改代码 |

## 六、已知约束与局限

| 约束 | 说明 |
|------|------|
| 无物理 NPU | 当前测试环境为 x86_64 CPU，AscendCL 运行时无法初始化（`acl.init()` 失败），使用 ATC 工具链验证替代 |
| `ascendhub.huawei.com` 不可达 | 官方文档给出的主要镜像地址在公网 DNS 不可解析，需使用 Docker Hub（`ascendai/cann`）替代 |
| Gitcode 社区仓库路径非直觉 | 贡献阶段目标仓库为 `cann/community`（非 `ascend/cann`），需通过浏览 [gitcode.com/cann](https://gitcode.com/cann) 确认 |
| 官方文档为 SPA | hiascend.com 文档为 Nuxt.js 单页应用，内容由 JS 动态渲染，自动化测试无法直接抓取正文 |
| Issue 提交为真实 issue | 贡献阶段提交的 issue 必须是真实发现的问题（非测试占位），否则无法测量社区真实响应速度 |

## 七、成功标准

### 7.1 框架层面

| 标准 | 判断方式 |
|------|---------|
| 四个阶段均可独立运行并输出指标 | 单元测试全部通过 |
| 报告包含所有必需字段 | 字段完整性测试通过 |
| token 为空时贡献阶段提前失败 | 专项测试验证 |
| 某阶段失败不影响后续阶段指标采集 | 集成测试验证 |

### 7.2 测试数据层面

| 指标 | 基准值（2026-03-26 首次测试） | 观测方向 |
|------|------------------------|---------|
| TTFHW（了解+获取+使用） | 约 30~38 分钟（手动）| 越小越好 |
| 镜像下载时间 | ~6 分钟（4.08 GB） | 越小越好 |
| Issue 首次响应时间 | 待观测（Issue #95） | 越小越好 |
| 官方镜像地址可用性 | ❌（`ascendhub.huawei.com` DNS 失败） | 期望改善为 ✅ |
| 文档地址稳定性 | ✅ | 保持稳定 |
