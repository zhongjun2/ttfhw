# CANN 易用性评估工具 — 需求分析

**版本：** v1.0
**创建日期：** 2026-03-26

---

## 一、背景与目标

### 用户场景

一个新用户想要在昇腾平台上使用 CANN 完成模型推理，他的实际操作路径是：

1. 在 Google 搜索 CANN 的相关信息
2. 根据搜索到的信息下载和安装 CANN
3. 根据社区提供的 Quick Start 或 README 跑通示例场景

### 评估目标

针对上面三步操作，构建一套验证方案和工具，量化评估：
- **各步骤实际耗时**（搜索、下载、安装、跑通示例）
- **操作易用性**（步骤数、能否顺利完成、是否需要绕路）
- **断点问题**（哪些步骤失败、为什么失败、如何解决）

输出结构化评估报告，供后续优化参考和持续回归对比。

## 二、测试范围

### 2.1 三个阶段

| 阶段 | 起点 | 终点 | 核心问题 |
|------|------|------|---------|
| **了解** | 打开 Google | 找到安装文档 + Quick Start + Qwen2 部署指引 | 官方资料能不能被搜到？链接是否有效？ |
| **获取** | 知道安装方式 | CANN 环境就绪可用 | Docker 和 .run 两种方式各需多久？有哪些坑？ |
| **使用** | CANN 环境就绪 | Quick Start 跑通 + Qwen2 命令可执行 | 工具链是否可用？示例能跑通吗？ |

### 2.2 获取阶段：两种安装方式

| 方式 | 命令 | 适用场景 |
|------|------|---------|
| **Docker 镜像** | `docker pull ascendai/cann:latest` | 快速体验，环境隔离 |
| **.run 包安装** | `./Ascend-cann-toolkit_*.run --install` | 裸机安装，接近生产环境 |

两种方式均测，分别记录耗时和步骤，输出对比。

### 2.3 使用阶段：两层验证

| 层次 | 内容 | 成功标准 |
|------|------|---------|
| **Quick Start** | ATC 工具链验证，CANN 基础功能可用 | `atc --help` 正常响应，基础示例命令无报错 |
| **Qwen2 推理** | 下载 Qwen2 模型、配置环境、执行推理命令 | 命令能启动、环境配置完成（CPU 机器：到命令启动无报错为止） |

> **CPU 机器说明：** 当前测试环境无物理 NPU，Qwen2 实际推理无法在 CPU 上运行。验证边界为：模型文件可下载、依赖包可安装、推理命令可启动（不因缺少驱动以外的原因报错）。

### 2.4 不在范围内

- 物理 NPU 上的推理性能指标（latency、throughput）
- 多机多卡分布式训练
- MindSpore / PyTorch_NPU 框架层面的调优

## 三、两种测试模式

### 模式 A：自动化测试

程序自动完成搜索、安装、执行，适合持续集成和定期回归。

```
python runner.py --mode auto --install docker
python runner.py --mode auto --install run_package
```

- 模拟搜索（调用搜索 API）
- 自动拉取 Docker 镜像或下载 .run 包
- 自动执行 Quick Start 命令
- 自动记录每步耗时，汇总输出报告

### 模式 B：人工辅助记录

真人操作，工具在旁边引导步骤、计时、收集断点信息，最终生成报告。

```
python manual/recorder.py
```

- 按预设步骤清单逐步提示
- 每步开始/结束时按 Enter 记录时间
- 支持输入断点备注（报错信息、需要绕路的操作）
- 测试结束后自动生成与自动化模式相同格式的报告

## 四、功能需求

### F1：了解阶段

| 编号 | 需求 | 优先级 |
|------|------|--------|
| F1.1 | 搜索 "CANN 昇腾 安装" 关键词，找到 hiascend.com 官方文档入口 | P0 |
| F1.2 | 找到 Quick Start 文档链接，验证 HTTP 可访问 | P0 |
| F1.3 | 找到 Qwen2 在 CANN/昇腾平台上的部署文档链接 | P0 |
| F1.4 | 记录搜索耗时、官方链接排名、文档可达性 | P0 |
| F1.5 | 检测找到的链接中的断链 | P1 |

### F2：获取阶段

| 编号 | 需求 | 优先级 |
|------|------|--------|
| F2.1 | Docker 方式：执行 `docker pull`，记录下载时间和镜像大小 | P0 |
| F2.2 | .run 方式：从 hiascend.com 找到下载链接，下载安装包 | P0 |
| F2.3 | .run 方式：执行 `./Ascend-cann-toolkit_*.run --install`，记录安装耗时和步骤数 | P0 |
| F2.4 | 两种方式均验证安装后工具链可用（`atc --help`） | P0 |
| F2.5 | 捕获并记录所有错误信息，标注为断点 | P0 |
| F2.6 | 输出两种方式的耗时对比 | P1 |

### F3：使用阶段

| 编号 | 需求 | 优先级 |
|------|------|--------|
| F3.1 | Quick Start：执行 ATC 基础示例（模型转换命令），验证工具链 | P0 |
| F3.2 | Qwen2：根据社区文档配置 MindSpore/torch_npu + CANN 环境 | P0 |
| F3.3 | Qwen2：下载 Qwen2 模型文件（HuggingFace 或 ModelScope） | P0 |
| F3.4 | Qwen2：执行推理命令，记录命令是否能启动、是否因非硬件原因报错 | P0 |
| F3.5 | 记录各步骤耗时和错误 | P0 |

### F4：报告输出

| 编号 | 需求 | 优先级 |
|------|------|--------|
| F4.1 | 输出 JSON 格式报告（机器可读，含所有指标字段） | P0 |
| F4.2 | 输出 Markdown 格式中文报告 | P0 |
| F4.3 | 报告包含：各阶段耗时、详细操作步骤、断点问题清单 | P0 |
| F4.4 | 断点问题按严重程度标注（P0 阻断 / P1 绕路可过 / P2 轻微） | P1 |
| F4.5 | 支持多次测试结果横向对比（相同步骤不同日期） | P1 |

### F5：接口

| 编号 | 需求 | 优先级 |
|------|------|--------|
| F5.1 | MCP 工具接口，可通过 Claude 触发测试和查看报告 | P0 |
| F5.2 | Claude Skill 命令（`/cann-eval`） | P0 |
| F5.3 | 命令行入口 `python runner.py` 独立运行 | P0 |

## 五、非功能需求

| 需求 | 说明 |
|------|------|
| **可重复性** | 相同环境多次运行结果可对比；Docker 方式屏蔽环境差异 |
| **无侵入性** | 不修改被测环境；测试完成后自动清理容器 |
| **敏感信息** | API key / token 通过环境变量注入，不写入代码或报告 |
| **容错性** | 单步失败记录断点后继续，不丢失其他步骤数据 |
| **可扩展** | 统一 Stage 接口，后续可新增其他安装方式或模型 |

## 六、报告结构

```
# CANN 易用性评估报告

## 总览
- 测试日期、环境、测试模式
- 各阶段状态（pass/warn/fail）和关键耗时

## 了解阶段
- 操作步骤记录
- 找到的链接列表（含可访问性）
- 断点问题

## 获取阶段
- Docker 方式：步骤 + 耗时 + 断点
- .run 方式：步骤 + 耗时 + 断点
- 两种方式对比

## 使用阶段
- Quick Start：步骤 + 耗时 + 断点
- Qwen2 推理：步骤 + 耗时 + 断点（含 CPU 限制说明）

## 断点汇总
- 所有断点按严重程度排列
- 每个断点：现象、原因、解决方法/绕路方案
```

## 七、约束与已知限制

| 约束 | 说明 |
|------|------|
| 无物理 NPU | Qwen2 实际推理无法执行；验证到"命令启动不因软件原因失败"为止 |
| hiascend.com SPA | 官方文档为动态渲染，自动化测试通过搜索 API 获取链接，不直接抓取页面正文 |
| .run 安装需 root | Ascend-cann-toolkit .run 安装默认需要 root 或 sudo；测试环境需提前确认权限 |
| Qwen2 模型较大 | Qwen2-7B 约 14GB，下载时间受网络影响大；测试时记录下载速度 |
| Docker Hub 替代 | 官方 `ascendhub.huawei.com` 公网 DNS 不可解析，实际使用 `ascendai/cann:latest` |
