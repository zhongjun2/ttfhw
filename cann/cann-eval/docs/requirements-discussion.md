## 需求澄清问答

### 问题一

**Q：测试执行方式：这套验证工具是全自动跑（程序模拟搜索、安装、推理），还是辅助真人操作（人在测，工具在旁边记录时间和步骤）？**

选项：
- 全自动：程序自动完成搜索、下载、安装、跑推理，适合持续集成和定期回归
- 辅助人工：真人操作，工具负责计时、记录步骤、捕获断点，最终生成报告，更贴近真实用户体验
- 两者都要：自动化测试用于日常监控，人工辅助测试用于深度体验评估

**A：两者都要**

---

**Q：测试环境：运行机器上有没有物理的昇腾 NPU 硬件？**

选项：
- 有 NPU 硬件：有昇腾 910/310 等硬件，可以真实跑推理
- 纯 CPU 机器：没有 NPU，需要通过 CPU 模拟或容器工具链验证
- 不确定，两种都要支持：工具需要自动检测环境，有 NPU 就跑真实推理，没有就用 CPU 验证工具链

**A：纯 CPU 机器**

---

**Q：获取方式：CANN 的安装方式以哪种为主？**

选项：
- Docker 镜像：docker pull 拉取官方镜像，环境隔离，最简单
- .run 包裸机安装：下载 Ascend-cann-toolkit_*.run 安装到宿主机，更贴近生产环境
- 两种都测：分别测两种安装方式的耗时和步骤数，做对比

**A：两种都测**

---

**Q：示例场景：模型推理跑通用哪个示例？**

选项：
- 官方 Quick Start 自带示例：直接跑 hiascend.com 快速入门里给的第一个示例，最贴近新手路径
- 指定模型（如 ResNet/BERT）：用特定模型做推理，结果更有业务参考价值，但步骤更复杂
- 先跑 Quick Start，再跑指定模型：分两层：最小验证 + 实际场景验证

**A：先跑 Quick Start，再跑指定模型**

---
